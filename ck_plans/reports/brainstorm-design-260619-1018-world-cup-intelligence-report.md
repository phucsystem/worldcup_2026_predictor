# World Cup Intelligence — Design Summary

**Date:** 2026-06-19 | **Status:** Approved, ready for `/ck:plan` | **Type:** Brainstorm → design

---

## Problem statement
AI-powered platform that auto-monitors FIFA World Cup 2026 and publishes a daily intelligence brief (completed matches, fixtures, standings, qualification scenarios, contenders) every morning at 7:00 AM, no manual intervention.

## Confirmed requirements (locked with user)
- **Primary goal:** Live product for the *current* WC 2026. Ship-fast, reliability-first.
- **Hard reality:** Tournament ends **July 19, 2026** → ~30-day live value window. Lasting asset = reusable pipeline + portfolio.
- **Data source:** Not pre-chosen → recommend below.
- **LLM:** **DeepSeek** only (cost). OpenAI-compatible API.
- **Infra:** **Azure firm, monorepo.** Container Apps + Azure Postgres.
- **Pipeline:** Keep **LangGraph** (satisfies showcase business goal).
- **Brief timezone:** **Australia/Melbourne**, 7:00 AM.
- **Frontend:** **React (Next.js App Router, SSR)** as a Container App.

## Expected output (V1)
1. Daily DB record in `articles` (title, summary, markdown body) generated by the agent pipeline.
2. Next.js (React + TS) SSR site: brief list + brief detail + standings tables + archive.
3. Azure Container Apps Job runs the pipeline on cron at 7 AM AEST/AEDT.

## Acceptance criteria
- Manual trigger produces one correct end-to-end brief in DB (Phase 1 gate).
- Standings/qualification numbers are **deterministically computed in Python** — LLM only narrates (zero table-math hallucination).
- Cron job publishes daily with no manual step; re-run is idempotent.
- `agent_runs` logs timings, tokens, cost, errors per run.

## Scope boundary (V1 OUT)
- Multi-LLM provider (GPT-5/Claude premium path) — deferred, behind same interface.
- V2/V3 agents: Prediction, Social, Tournament Director, Stats, News.
- Live (sub-minute) scores — daily delayed data is sufficient.

## Architecture

### Data source
- **Primary: API-Football (API-Sports) free tier** — 100 req/day (≫ needs), fixtures + match *events* + standings + groups + knockout bracket. Richer events → better summaries.
- **Thin `DataSource` interface, one impl.** football-data.org (free forever, WC covered) documented as drop-in fallback. **Not** building two integrations.
- **Seed** groups/schedule once from `openfootball/worldcup.json` (free static, no key).

### Agent pipeline (LangGraph linear StateGraph, 3 nodes, shared typed state)
```
[Collector] → [Analyst] → [Editor]
 no LLM        DeepSeek      DeepSeek
 fetch+        structured    markdown article
 normalize+    intel JSON    + title + summary
 compute math
```
- **Collector (no LLM):** fetch via DataSource, normalize, persist `matches`/`standings`, **compute points/GD/position deltas/qualification math in Python.**
- **Analyst (DeepSeek):** consumes pre-computed facts → structured intelligence JSON (storylines, surprises, underperformers, power ranking). Never does arithmetic.
- **Editor (DeepSeek):** intelligence JSON → markdown article + title + summary.
- LangGraph chosen for: showcase goal, clean state passing, built-in checkpoint/retry. Honest note: plain pipeline would also work; kept for the stated showcase requirement and V2 headroom (conditional branches/retries).

### Scheduling
- **Azure Container Apps Job**, cron-scheduled, 7 AM Australia/Melbourne (handle DST). Separate from always-on API container. No bespoke scheduler.

### Monorepo layout
```
/backend     FastAPI read API + LangGraph pipeline + collector
/frontend    Next.js (React + TypeScript) + Tailwind (SSR)
/infra       Azure Bicep / container config
/db          migrations
docker-compose.yml   local dev
```

### Deployment topology (Azure)
- Container App: FastAPI read API (always-on).
- Container Apps **Job**: cron pipeline.
- Container App: Next.js (React + TS) SSR (reads FastAPI).
- Azure Database for PostgreSQL Flexible Server.

### DB schema
- `matches` (id, group, home/away, scores, status, kickoff, events_json)
- `standings` (snapshot_date, group, team, P/W/D/L/GF/GA/GD/pts, position, prev_position)
- `articles` (date, title, summary, body_md, status, model_used)
- `agent_runs` (run_id, started/finished, node timings, tokens, cost, status, error)

## Phasing (30-day ship-fast)
- **Phase 1 — vertical slice:** manual-trigger pipeline → one brief in DB → bare Next.js page renders it. Prove chain before automating.
- **Phase 2 — schedule + reliability:** Container Apps Job cron, `agent_runs` logging, retries, idempotency.
- **Phase 3 — polish:** Next.js UI (list/detail/standings/archive), deploy hardening.

## Key risks & mitigations
- **LLM table-math hallucination** → compute all standings/qualification math in Python; LLM narrates only. (Highest-priority correctness control.)
- **Free-tier data access blocks current WC season** → fallback to football-data.org behind `DataSource` interface.
- **Short live window (ends July 19)** → optimize Phase 1 for shipping a correct brief *this week*; treat architecture polish as secondary.
- **DST on 7 AM cron** → use TZ-aware scheduling (Australia/Melbourne).

## Decisions cut from original brief
- 3 LLM providers → 1 (DeepSeek).
- GPT-5/Claude premium path → V2.
- V2/V3 multi-agent expansion → out of scope, schema leaves room.
